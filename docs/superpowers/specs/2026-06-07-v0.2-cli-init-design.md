# v0.2 CLI init design

- Date: 2026-06-07
- Scope: `@bridgent/cli` onboarding command, docs, and tests
- Trigger: v0.2 roadmap prioritizes onboarding before adding more source adapter surface.

## 1. Background

v0.1 requires users to already know how to write a Bridgent server file. That is acceptable for alpha users reading the docs, but it creates avoidable friction for first-run CLI usage.

The v0.2 roadmap calls for `bridgent init` before `bridgent expose`. The important product rule is that init should generate a normal server file users can read and edit. It should not introduce a hidden project model or configuration layer.

## 2. Goals

Add a focused `bridgent init` command that:

1. Generates a minimal `server.ts` using `@bridgent/core`, Zod, and `createStdioServer`.
2. Protects existing files by default.
3. Supports an explicit overwrite path for repeatable setup.
4. Prints concise next-step commands for development and install hints.
5. Keeps the generated file compatible with Node 22.18 type stripping.

## 3. Non-goals

- No `bridgent expose` command.
- No generated config format.
- No package-manager mutation or dependency installation.
- No project scaffold beyond the selected server file.
- No source adapter generation for OpenAPI, Prisma, Drizzle, tRPC, or GraphQL.
- No custom inspector changes.

## 4. Command Shape

```bash
bridgent init [file] [--force]
```

Defaults:

- `file`: `server.ts`
- `--force`: false

Behavior:

- Resolve the target relative to the current working directory.
- Create parent directories recursively.
- If the target exists and `--force` is not set, exit with code 1 and explain how to overwrite.
- Write a readable starter server with one `echo` tool.
- Print:
  - created path
  - packages to install if missing: `@bridgent/cli`, `@bridgent/core`, `zod`
  - next commands: `bridgent dev <file>` and `bridgent inspect <file>`

## 5. Generated Server

The generated file should use direct imports:

```ts
import { createStdioServer, defineTool } from '@bridgent/core'
import { z } from 'zod'
```

It should define one `echo` tool and start stdio:

```ts
await createStdioServer({
  name: 'bridgent-starter',
  version: '0.1.0',
  tools: [echo],
})
```

Use a local ESLint disable comment for top-level await, matching the existing example convention.

## 6. Test Strategy

Add CLI unit tests around pure helper functions instead of spawning the CLI:

- default target is `server.ts`
- custom nested target creates parent directories
- existing file is protected by default
- `force` overwrites
- generated content includes imports, tool name, and `createStdioServer`

No network or package install is needed for these tests.

## 7. Documentation

Update the public getting-started docs in English and Chinese to show:

```bash
pnpm dlx @bridgent/cli init server.ts
bridgent dev server.ts
```

Keep manual server authoring visible so users understand the runtime shape.

## 8. Validation

Required validation:

- `pnpm --filter @bridgent/cli test`
- `pnpm --filter @bridgent/cli build`
- `pnpm --filter @bridgent/cli typecheck`
- `pnpm --filter @bridgent/docs lint`
- `pnpm --filter @bridgent/docs build`
- Full gate when feasible: `pnpm turbo run build test typecheck lint`
